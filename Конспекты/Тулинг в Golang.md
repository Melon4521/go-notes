## Что это и зачем?

**Тулинг** (tooling) в Go - это набор встроенных инструментов, которые помогают выполнять второстепенную работу по оформлению, проверке и очистке кода.

Есть как и встроенные в Go инструменты (`go fmt`, `go test`, `go vet`, `go doc`), так и внешние (`gopls`, `goimports`, `golangci-lint`, `govulncheck`).

Установить внешние инструменты можно через `go install`:

```bash
go install golang.org/x/tools/gopls@<version>
go install golang.org/x/tools/cmd/goimports@<version>
go install golang.org/x/vuln/cmd/govulncheck@<version>
```

## Форматирование

Работу форматирования выполняет утилита `gofmt`. У нее есть верхнеуровневая обертка `go fmt`, которая является упрощенным вызовом `gofmt` (`gofmt -l -w` для всех `.go` файлов в пакете). А сама утилита `gofmt` - универсальный инструмент форматирования для конкретных файлов (или `stdin`). У нее есть специальные флаги:
- `-w` - без этого флага форматирование не запишется в файл
- `-l` - выводит список файлов, которые не соответствуют стандартному форматированию
- `-d` - не меняет файл, а показывает разницу текущего и будущего, отформатированного
- `-s` - пытается упростить код (уберет лишние преобразования типов, заменит громоздкие конструкции на более простые и т.д)
- `-r` - мощный флаг для замены одних выражение на другие. Пример: `gofmt -r 'a[b:len(a)] -> a[b:]' -w .` - удалит лишние `len` во всех слайсах проекта

**Использование:**
- Можно отформатировать текущую директорию и все вложенные файлы через `go fmt ./...`
- Отформатировать конкретный файл - `gofmt -w file.go`, весь проект - `gofmt -w .`

Для более продвинутого форматирования с обновлением импортов есть утилита `goimports` - она имеет те же флаги, что и у `gofmt`, но дополнительно к форматированию кода еще и добавляет нужные / удаляет ненужные импорты.

## Очищение кода

Инструмент `go vet` позволяет нам избежать наиболее частые и опасные шаблоны кода, которые ведут к логическим ошибкам. Например:
- Проверка бесполезных объявлений переменных
- Проверка типичных ошибок при работе с пакетом `sync/atomic`
- Проверка неиспользуемого кода и т.д.

Запустить для всего проекта: `go vet ./...`

## Линтеры

**Линтеры** - это класс инструментов статического анализа кода, которые автоматически проверяют исходный код на наличие ошибок и стилевых нарушений. По сути, инструменты выше ими являются. Но иногда нам необходимы более строгие правила написания кода - для этого есть утилита `golangci-lint`, которая по сути является набором линтеров.

Настроить линтинг можно через `yaml`-файл конфигурации. К примеру, можно добавить следующие проверки:
- проверка на ограничение длины строки
- проверка на цикломатическую сложность
- проверка на длину функции и т.д

Пример файла конфигурации:

```yaml
run:
  timeout: 5m
  modules-download-mode: readonly
linters:
  disable-all: true # Отключить всё по умолчанию
  enable:          # Что включено принудительно
    - errcheck     # Проверка необработанных ошибок
    - goimports    # Проверка импортов
    - golint       # Рекомендации по стилю (устаревает, но полезен)
    - govet        # Тот самый go vet
    - staticcheck  # Статический анализ
    - unused       # Неиспользуемый код
    - gosimple     # Упрощение кода
    - bodyclose    # Закрытие http-тел
    - dupl         # Поиск дублирования кода
    - reassign     # Поиск переопределения переменных
    - unconvert    # Лишние приведения типов
issues:
  exclude-use-default: false # Не использовать стандартные исключения
  max-issues-per-linter: 0   # Показать все ошибки
  max-same-issues: 0         # Показать все похожие ошибки
  exclude-rules:
    - path: _test\.go
      linters:
        - goconst    # Игнорировать дублирование строк в тестах
```

## Проверка зависимостей

Утилита `govulncheck` - проверяет используемые в проекте зависимости на предмет известных уязвимостей: `govulncheck ./...`. По сути, утилита сканирует текущий проект, анализируя список в `go.mod`. И не просто сканирует, проверяя, есть ли в импортах уязвимые библиотеки, а смотрит, используется ли уязвимый код.

## Набор вызовов для полной обработки кода

**Пайплайн такой:**
1. Приводим в порядок зависимости
2. Форматируем код
3. Базовая проверка на типичные ошибки
4. Linter
5. Проверка безопасности
6. Тесты (обычные + с детектором гонок)

```bash
go mod tidy
go mod verify

go fmt ./...

go vet ./...

golangci-lint run

govulncheck ./...

go test -v -race -cover ./...
```
