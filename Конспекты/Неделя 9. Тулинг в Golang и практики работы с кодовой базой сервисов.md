# Тулинг в Golang

## Что это и зачем?

**Тулинг** (tooling) в Go - это набор встроенных инструментов, которые помогают выполнять второстепенную работу по оформлению, проверке и очистке кода.

Есть как и встроенные в Go инструменты (`go fmt`, `go test`, `go vet`, `go doc`), так и внешние (`gopls`, `goimports`, `golangci-lint`, `govulncheck`).

Установить внешние инструменты можно через `go install`:

```bash
go install golang.org/x/tools/gopls@<version>
go install golang.org/x/tools/cmd/goimports@<version>
go install golang.org/x/vuln/cmd/govulncheck@<version>
```

## Форматирование

Работу форматирования выполняет утилита `gofmt`. У нее есть верхнеуровневая обертка `go fmt`, которая является упрощенным вызовом `gofmt` (`gofmt -l -w` для всех `.go` файлов в пакете). А сама утилита `gofmt` - универсальный инструмент форматирования для конкретных файлов (или `stdin`). У нее есть специальные флаги:
- `-w` - без этого флага форматирование не запишется в файл
- `-l` - выводит список файлов, которые не соответствуют стандартному форматированию
- `-d` - не меняет файл, а показывает разницу текущего и будущего, отформатированного
- `-s` - пытается упростить код (уберет лишние преобразования типов, заменит громоздкие конструкции на более простые и т.д)
- `-r` - мощный флаг для замены одних выражение на другие. Пример: `gofmt -r 'a[b:len(a)] -> a[b:]' -w .` - удалит лишние `len` во всех слайсах проекта

**Использование:**
- Можно отформатировать текущую директорию и все вложенные файлы через `go fmt ./...`
- Отформатировать конкретный файл - `gofmt -w file.go`, весь проект - `gofmt -w .`

Для более продвинутого форматирования с обновлением импортов есть утилита `goimports` - она имеет те же флаги, что и у `gofmt`, но дополнительно к форматированию кода еще и добавляет нужные / удаляет ненужные импорты.

## Очищение кода

Инструмент `go vet` позволяет нам избежать наиболее частые и опасные шаблоны кода, которые ведут к логическим ошибкам. Например:
- Проверка бесполезных объявлений переменных
- Проверка типичных ошибок при работе с пакетом `sync/atomic`
- Проверка неиспользуемого кода и т.д.

Запустить для всего проекта: `go vet ./...`

## Линтеры

**Линтеры** - это класс инструментов статического анализа кода, которые автоматически проверяют исходный код на наличие ошибок и стилевых нарушений. По сути, инструменты выше ими являются. Но иногда нам необходимы более строгие правила написания кода - для этого есть утилита `golangci-lint`, которая по сути является набором линтеров.

Настроить линтинг можно через `yaml`-файл конфигурации. К примеру, можно добавить следующие проверки:
- проверка на ограничение длины строки
- проверка на цикломатическую сложность
- проверка на длину функции и т.д

Пример файла конфигурации:

```yaml
run:
  timeout: 5m
  modules-download-mode: readonly
linters:
  disable-all: true # Отключить всё по умолчанию
  enable:          # Что включено принудительно
    - errcheck     # Проверка необработанных ошибок
    - goimports    # Проверка импортов
    - golint       # Рекомендации по стилю (устаревает, но полезен)
    - govet        # Тот самый go vet
    - staticcheck  # Статический анализ
    - unused       # Неиспользуемый код
    - gosimple     # Упрощение кода
    - bodyclose    # Закрытие http-тел
    - dupl         # Поиск дублирования кода
    - reassign     # Поиск переопределения переменных
    - unconvert    # Лишние приведения типов
issues:
  exclude-use-default: false # Не использовать стандартные исключения
  max-issues-per-linter: 0   # Показать все ошибки
  max-same-issues: 0         # Показать все похожие ошибки
  exclude-rules:
    - path: _test\.go
      linters:
        - goconst    # Игнорировать дублирование строк в тестах
```

## Проверка зависимостей

Утилита `govulncheck` - проверяет используемые в проекте зависимости на предмет известных уязвимостей: `govulncheck ./...`. По сути, утилита сканирует текущий проект, анализируя список в `go.mod`. И не просто сканирует, проверяя, есть ли в импортах уязвимые библиотеки, а смотрит, используется ли уязвимый код.

## Набор вызовов для полной обработки кода

**Пайплайн такой:**
1. Приводим в порядок зависимости
2. Форматируем код
3. Базовая проверка на типичные ошибки
4. Linter
5. Проверка безопасности
6. Тесты (обычные + с детектором гонок)

```bash
go mod tidy
go mod verify

go fmt ./...

go vet ./...

golangci-lint run

govulncheck ./...

go test -v -race -cover ./...
```



# Идиоматичный код в Go

## Идиомы

**Идиомы языка** - список правил и приемов для написания правильного кода на Go. Идиомы в Go направлены на **читаемость**, **простоту** и **явность**.

Полезные материалы:
- Основной список идиом языка: [Go Proverbs](https://go-proverbs.github.io/)
- Гайд "Эффективный Go": [Effective Go](https://go.dev/doc/effective_go)
- Дополнение к Effective Go: [Go Code Review Comments](https://go.dev/wiki/CodeReviewComments)

## Наименование пакетов

Рекомендуется называть пакеты максимально кратко, чтобы не усложнять себе жизнь при импортах.

Если название пакета содержит 2 слова, то разделять их `_` не надо: `package userinfo`

Также стоит выбирать такие названия пакетов, чтобы их наименования не совпадали с именами переменных в импортируемых пакетах, в ином случае придется использовать alias пакета:

```go
import (
  usersrv github.com/..../user
)
func Do() {
  user, err := usersrv.TryFind() // переменная user и пакет user
}
```

Еще одно важное правило - стоит давать осмысленные имена пакетам, не стоит называть их обезличено (`utils`, `common`, `helper`, `model` ...). Хорошие примеры: `email`, `auth`, `config`

## Названия методов и функций

Важное правило - избегать дублирования имени пакета в именах его сущностей:

**Плохой пример:**

```go
package db

func CreateDbConnection() (*DbConn, error) {}
```

Дублирование в использовании: `db.CreateDbConnection`, `db.DbConn`.

**Хороший пример:**

```go
package db

type Conn struct {}

func CreateConn() (*Conn, error) {}
```

Где-то в другом пакете:

```go
c, err := db.CreateConn()

func Do(c *db.Conn) {}
```


Также в Go стоит избегать функции-геттеры с префиксом `get`:

```go
package user

type User struct {
  Id int
  Address string
}

// Плохой пример
// Использование: address := user.GetAddress()
func (u User) GetAddress() string {
  return u.Address
}

// Хороший пример
// Использование: address := user.Address()
func (u User) Address() string {
  return u.Address
}
```

Тут есть небольшое замечание: в пакете `user` есть структура `User`, она полностью дублирует название пакета и по сути является главной сущностью пакета. Это нормально.

## Интерфейсы

Если интерфейс имеет единственный метод, то его называют по его именованию:
- `Read()` -> `Reader`
- `Write()` -> `Writer`

Интерфейс стоит объявлять там, где он используется, а не в месте, где реализуются его методы:

```go
package format

type Formatter interface {
  Format() string
}

type Service struct {}

func NewService() *Service {
  return &Service{}
}

func (s *Service) Format() string {}
```

Где-то в другом пакете будет не важно, что мы используем: метод напрямую или интерфейс. Мы все равно притянем пакет format

```go
package example

import github.com/..../format

func example(f format.Formatter) {
  svc := format.NewService()
  svc.Format()
  f.Format()
}
```

Лучше всё же вынести интерфейс в место использования. Так не получим лишних импортов и полностью абстрагируем логику

```go
package example

type Formatter interface {
  Format() string
}

func example(f Formatter) {
  f.Format()
}
```

## nil-мапы и слайсы

Не стоит проверять мапы и слайсы на то, что они неинициализированы сравнением с `nil`. Достаточно использовать `len()`, т.к она вернет в таком случае 0.

## Горутины

Когда запускаем горутину, нужно сразу же понять, где и при каких обстоятельствах она закроется. 

Если без горутины можно обойтись, лучше так и сделать.

## Каналы

По умолчанию всегда стоит делать каналы небуферизированными, и только в случае дальнейшей необходимости - объявляем с буфером.

## Параметры функции

Вместо подобной сигнатуры с кучей параметров:

```go
func Example(one int, two string, three uint, four, five uuid.UUID, six any, seven string, eight bool) (map[int]string, uint, []int, any, error) {}
```

Лучше использовать отдельную структуру для параметров и для возврата в сигнатуре:

```go
type ExampleRequest struct {
  one int
  two string
  three uint
  four, five uuid.UUID
  six any
  seven string
  eight bool
}

type ExampleResp struct {...}

func Example(req ExampleRequest) (ExampleResp, error) {}
```

## Ошибки и паники

Всегда стоит обрабатывать ошибки и оборачивать их в контекст через `fmt.Errorf` при возврате.

Ну и конечно, не стоит злоупотреблять вызовом паник.

## Дженерики

Дженерики стоит использовать только там, где они действительно необходимо. Использовать их с мыслью "код универсальный, пригодится в будущем" неправильно.

## Глобальные переменные

Никаких глобальных переменных по возможности.
