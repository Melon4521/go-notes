**REST** - архитектурный стиль построения API веб-серверов с определенным набором правил.

## Принципы REST

- **Клиент-серверная архитектура** - клиент отвечает за использование API, а сервер - за правильную работоспособность и соответствие стилю
- **Отсутствие состояния** - сервер не хранит данные о сессии клиента и каждый запрос является независимым
- **Кеширование** - ответы от сервера должны быть помечены как кешируемые или нет, чтобы клиент мог повторно использовать полученные данные
- **Единообразие интерфейса** - способ взаимодействия через HTTP-методы, понятие ресурсов и URI
- **Многослойная система** - разделение компонентов на отдельные уровни (балансировщик нагрузки, HTTP-обработчик), которые не могут видеть дальше какого-то другого уровня.
- **Код по требованию** (необязательный пункт) - возможность расширять функциональность клиентов за счёт загрузки кода с сервера (например, передача клиенту каких-то скриптов).

**REST API** в серверах реализует операции **CRUD** - create, read, update, delete - с помощью методов POST, GET, PUT/PATCH, DELETE соответственно.
## Понятие ресурса

**Ресурс** - любая сущность данных, имеющая при себе **уникальный id**, по которому к ресурсу можно обратиться через URI. Например, товар, пользователь и т.д.

```
Хорошо: ресурсно-ориентированные URI

GET    /api/users          — Получить всех пользователей
GET    /api/users/123      — Получить пользователя с ID=123
POST   /api/users          — Создать нового пользователя
PUT    /api/users/123      — Обновить пользователя 123
DELETE /api/users/123      — Удалить пользователя 123

GET    /api/users/123/orders     — Заказы пользователя 123
GET    /api/orders/123/items     — Позиции в заказе 123

ПЛОХО: глаголы в URI

GET    /api/getUsers        — Неправильно!
POST   /api/createUser      — Неправильно!
PUT    /api/updateUser/123  — Неправильно!
```
