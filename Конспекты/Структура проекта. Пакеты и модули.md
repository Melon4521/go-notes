## Структура проекта

Структура проекта в Go разделяется на **модули** и **пакеты**.

**Пакет** - это набор файлов (в одной папке) с одинаковым объявлением `package`. Пакеты в Go нужны, чтобы организовать структуру кода и сделать его переиспользуемым. 

**Модуль** - коллекция пакетов, определяется наличием файла `go.mod` в корневой папке модуля. Обычно модуль является полноценной работающей программой, т.е это наш проект.

Примерная структура для проекта на Go:

```
my-app/                 # Корень проекта
│
├── go.mod              # Модуль и зависимости
├── go.sum              # Хеши зависимостей
│
├── cmd/                # Точки входа (main пакеты)
│   ├── server/         # Основное приложение (сервер)
│   │   └── main.go
│   └── cli/            # Консольная утилита для этого же проекта
│       └── main.go
│
├── internal/           # Приватный код (нельзя импортировать снаружи)
│   ├── app/            # Логика инициализации приложения
│   ├── config/         # Чтение конфигов
│   ├── domain/         # Бизнес-сущности и интерфейсы
│   ├── service/        # Бизнес-логика
│   └── transport/      # Обработчики (HTTP/gRPC)
│
├── pkg/                # Публичный код (можно использовать в других проектах)
│   ├── logger/         # Кастомная обертка над логгером
│   └── validator/      # Утилиты для валидации
│
├── api/                # Спецификации (Swagger, Proto-файлы)
├── configs/            # Файлы конфигурации (.yaml, .env)
├── deployments/        # Docker-файлы, Kubernetes манифесты
├── scripts/            # Bash-скрипты для сборки или миграций
├── test/               # Дополнительные внешние тесты (e2e)
└── Makefile            # Команды для сборки и запуска
```

Подробнее про папки `cmd`, `internal` и `pkg`:
- `cmd` - папка с точками входа
- `internal` - содержит в себе пакеты, доступные только внутри нашего модуля, никто другой не сможет импортировать отсюда пакеты в свой проект.
- `pkg` - здесь находятся общедоступные пакеты, которые могут быть полезны для использования в других пакетах

## Пакеты

Каждый файл в Go начинается с указания пакета. Например: `package config` (у нас может быть сколько угодно файлов с объявлением такого пакета). При сборке проекта компилятор склеит все файлы из одного пакета в один большой файл.

Есть особое название для пакета - `main`. Файлы с `package main` - точка входа в программу. Среди всех файлов в пакете `main` обязана быть единственная функция `main()`, которая будет вызываться при запуске программы. 

> [!important]
> В Go принято называть пакет так же, как и папку, в которой находятся файлы пакета. Например, у нас есть папка `config`, в которой находятся файлы одного пакета, тогда его стоит обозвать именно `package config`.

> [!warning]
> В рамках одной папки не может быть два файла с разным указанием пакета `package <имя_пакета>`, это ошибка!

## Импорт пакетов

Чтобы использовать функции из пакетов необходимо их импортировать. Импорты принято делить на 3 группы:

```go
package main

import (
	// 1. Пакеты стандартной библиотеки Go
	"fmt"
	"net/http"
	
	// 2. Внешние пакеты
	"github.com/google/uuid"
	
	// 3. Внутренние пакеты в проекте (начинаются с названия модуля)
	"my-app/internal/config"
	"my-app/pkg/utils"
)

func main() {
	// ...
}
```

Для импорта определенного пакета нужно указать путь к папке, в котором лежат файлы этого пакета. Путь может быть **относительным**, как `fmt`, используем для стандартных пакетов, и **абсолютным**: используется для внешних пакетов (сторонние либы, скачиваем с помощью команды `go get`) и для внутренних пакетов проекта.

> [!warning]
> Циклические импорты в Go запрещены. Пакет `A` не может импортировать пакет `B`, когда пакет `B` импортирует пакет `A`. Для разрещения такого конфликта можно создать отдельный общий пакет `C` и импортировать его.

#### Дополнение к импортам

**1. Псевдонимы пакетов**: Если к примеру название пакета длинное, то его можно переобозначить и использовать новое имя для функций пакета:

```go
import (
	crand "crypto/rand"
	mrand "math/rand"
)

mrand.SomeFunction()
```

**2. Импорт с `_`**: Если нам не нужны функции пакета, но нам важно выполнение его функции `init()`, то можно указать `_` перед путем:

```go
import _ "github.com/lib/pq" // Регистрирует драйвер PostgreSQL
```

**3. Импорт с `.`**: Если мы хотим использовать функции пакета напрямую, без обращение через название пакета, то можно указать `.` перед путем:

```go
import . "fmt"

func main() {
    Println("Hello") // Вместо fmt.Println
}
```

## Области видимости пакетов

Здесь все те же правила:
- У функции/переменной/типа название начинается **с заглавной буквы** - этот объект экспортирован и может использоваться из других пакетов.
- У объекта название начинается **с маленькой буквы** - он не экспортирован и может использовать только внутри текущего пакета, не из других.

## Функция `init()`, инициализация пакетов

Внутри пакета может быть функция `init()`. Она выполнится на этапе инициализации проекта, если пакет вошел в сборку (используется где-либо в импортах, то есть в целом от него зависит код в `main` пакете).

Порядок инициализации пакета выглядит следующим образом:
1. Сначала импортируются все используемые пакеты
2. Инициализируются глобальные переменные на уровне пакета
3. Выполняются функции `init()` (в рамках пакета их может быть несколько, в таком случае порядок выполнения - алфавитный по названиям файлов, в которых есть `init()`)
   
После инициализации всех пакетов, используемых в программе, будет инициализирован сам пакет `main` и запущена функция `main()` в точке входа.

## Модули

Как было сказано выше, **модуль** - это коллекция пакетов, которая по большей части является полноценной программой.

Каждый проект в Go начинается с команды инициализации модуля):

```bash
go mod init <название_модуля>
```

После Go создает два файла:
- `go.mod` - файл описания модуля и зависимостей
- `go.sum` - файл с хешами зависимостей

## Файл `go.mod`

Пример файла `go.mod`:

```go.mod
module github.com/username/project-name

go 1.25

require (
    github.com/gin-gonic/gin v1.9.1
    github.com/golang-migrate/migrate/v4 v4.16.2
    github.com/lib/pq v1.10.9
    github.com/spf13/viper v1.16.0
    gorm.io/driver/postgres v1.5.2
    gorm.io/gorm v1.25.4
)

require (
    github.com/bytedance/sonic v1.9.1 // indirect
    github.com/chenzhuoyu/base64x v0.0.0-20221115062448-fe3a3abad311 // indirect
    // ... другие косвенные зависимости
)
```

#### Секции файла `go.mod`

**1. `module <название_модуля>`** - определяем **имя модуля**
Есть 2 варианта названия:
- С указанием домена, например: `github.com/username/project` - для публичных проектов и библиотек. Нужно указывать существующий адрес репозитория (необязательно из github), чтобы другие могли скачать проект через `go get`.
- Без домена, например `my-private-app` - для приватных/локальных проектов. Но даже для таких проектов принято добавлять домен в название.

**2. `go <версия_Go>`** - определяет минимальную версию Go для сборки проекта

**3. `require`** - определяет набор **прямых и косвенных зависимостей**.
**Прямые зависимости** - пакеты, которые непосредственно импортируются в коде нашего проекта, а **косвенные** (помечаются `// indirect`) - которые необходимы для работы прямых зависимостей.

```go.mod
require (
	// прямые:
    github.com/gin-gonic/gin v1.9.1
    github.com/golang-migrate/migrate/v4 v4.16.2
    // косвенные:
    github.com/bytedance/sonic v1.9.1 // indirect
    github.com/chenzhuoyu/base64x v0.0.0-20221115062448-fe3a3abad311
)
```

**4. `replace`** - для подмены одних зависимостей другими

```go.mod
replace github.com/old/package => github.com/new/package v1.2.3
replace github.com/local/package => ../local/package
```

**5. `exclude`** - для указания зависимостей, которые нужно пропустить и не загружать

```go.mod
exclude github.com/problematic/package v1.5.0
```

## Файл `go.sum`

Пример файла `go.sum`:

```go.sum
github.com/gin-gonic/gin v1.9.1 h1:4idEAncQnU5cB7BeOkPtxjfCSye0AAm1R0RVIqJ+Jmg=
github.com/gin-gonic/gin v1.9.1/go.mod h1:hPrL7Yrp6KXt5YId3A/Tnip5kqbEAP+KLuI3SUcPTeU=
github.com/bytedance/sonic v1.9.1 h1:6iJ6NqdoxCDr6mbY8h18oSO+cShGSMRGCEo7F2+0Y8=
github.com/bytedance/sonic v1.9.1/go.mod h1:i736AoUSYt75HyZLuJZ0SSRlHkmaJ2Kl9N6Fz5rD3k=
```

Этот файл необходим для безопасности: при скачивании зависимостей Go сверяет хеши зависимостей. Это позволяет предотвратить подмену зависимостей на вредоносный код.

Этот файл обновляется автоматически.

## Работа с зависимостями

- `go get <путь>` - Скачивание последней версии пакета, добавление в `go.mod`.
- `go get <путь>@v1.9.1` - Добавление конкретной версии.
- `go get <путь>@latest` - Обновление пакета до последней стабильной версии.
- `go get -u ./...` - Обновление всех зависимостей (юзать с осторожностью).
- `go mod tidy` - Умная команда для корректировки и удаления зависимостей проекта.
- `go mod vendor` - Копирует все внешние зависимости прямо в папку проекта.
- `go mod download` - Скачивает все зависимости из `go.mod` в кэш.
- `go clean -modcache` - Полностью очистить кэш скачанных модулей (если всё сломалось).
