## Файлы конфигураций

В cli-приложения начальные условия для работы можно устанавливать с помощью флагов. В остальных приложениях это удобнее всего сделать с помощью конфигураций.

## JSON

Сначала необходимо определить два понятия:
- **Сериализация** - процесс преобразования объекта/структуры языка, с которым мы работаем, в формат, который можно передать куда-либо и преобразовать обратно позже (`struct` -> `JSON`)
- **Десериализация** - обратный процесс, когда мы из общего формата хранения, переводим данные в структуры языка, чтобы работать с ними (`JSON` -> `struct`)

**JSON** - как раз такой формат для хранения сериализованных данных. Он представляет из себя объекты с парами `ключ: значение`. Значения могут быть строками, числами, `null`, булевыми значениями, массивами или вложенными `json`-объектами. Пример `json`-файла:

```json
[
    {
        "name": "Андрей",
        "age": 33,
        "isEmployed": true
    },
    {
        "name": null,
        "age": 21,
        "isEmployed": false
    }
]
```

## YAML

**YAML** - еще один формат хранения сериализованных данных в виде пар `ключ: значение` (чаще именно для хранения конфигурации), синтаксис которого проще чем у **JSON**: иерархия задается отступами. **YAML** можно преобразовать в **JSON** и обратно. В отличии от **JSON**, в **YAML** поддерживаются комментарии. Пример `yaml`-файла:

```yaml
server:
    port: 8080
    host: localhost    # строку можно писать и без кавычек
app:
    log_level: 2
    enabled_test: true
test: null
dev: ["local", "stage", "prod"]
```

Тип данных поля можно указать явно с помощью `!!type` перед значением:

```yaml
this_is_string: !!str 67.43
```

В **YAML** удобный способ переиспользования данных с помощью **якорей** `&` и **алиасов** `*`. Перед объектом, который мы хотим в дальнейшем переиспользовать мы ставим якорь с именем `&<имя_якоря>`, после можем вставлять его в другие поля через оператор слияния `<<:` и указания алиаса на якорь `*<имя_якоря>`:

```yaml
# заякорили значение поля default - это объект с тремя полями
default: &default_settings
    port: 8080
    timeout: 30
    protocol: https

local:
    <<: *default_settings # "распаковали" якорь внутрь объекта под ключом local
    host: localhost
    
# ------ результат в local после склейки ------

local:
    port: 8080
    timeout: 30
    protocol: https
    host: localhost
```

## Теги структур и работа с форматами данных

В Golang десериализованные данные являются структурами, так как это самый удобный формат хранения данных. Для указания связи поля структуры и какого-либо поля из **JSON**/**YAML** нам нужно указывать **теги структуры** внутри обратных кавычек после значения поля:

```go
type Config struct {
	Port int    `json:"port" yaml:"port"`
	Host string `json:"host" yaml:"host"`
}
```

Для работы с **JSON** в Golang есть пакета `encoding/json`. Он по умолчанию работает только с **экспортируемыми** полями структуры и в качестве названия для `json`-ключа берет либо само название поля, либо название из `json`-тега у этого поля.

Но `json`-теги могут быть очень гибкими:

```go
type User struct {
	// обычный случай
    Name     string `json:"name"`   
    
	// разные имена у поля и тега
    Surname  string `json:"last_name"`
    
	// если поля нет в json, то при десериализации оно будет иметь дефолтное значение типа в итоговой структуре. omitempty работает уже при сериализации и если в структуре у этого поля стоит дефолтное значение, то в json-файл это поле вообще не будет создано
    Age      int    `json:"age,omitempty"`
     
	// поле полностью исключается из рассмотрения
    Password string `json:"-"`
    
    // наименование ключа по названию поля (будет Address)
    Address  string `json:",omitempty"`
    
    // Наименование ключа по названию поля. string указывает на явное преобразование значения поля в строку
    Phone    string `json:",string"`        
    
    // похож на omitempty, но лучше работает с значениями по умолчанию, например, в случае с временными метками time.Time
    HasJob   bool   `json:"has_job,omitzero"`
    
    // указатель помогает присвоить полю структуры nil в случае, если в json-файле нет такого поля или оно null
    Foo      *int   `json:"foo"`
}
```

Пакет `encoding/json` позволяет сериализовать в **JSON** с помощью `bytes, err := json.Marshal(structure)`, а десериализовать через `err := json.Unmarshal(bytes, &structure)`:

```go
response := answer{Key: "value"}

// преобразуем go-объект в json-строку (байты)
bytes, err := json.Marshal(response)
if err != nil { log.Fatal(err) }

fmt.Println(string(bytes)) // {"key":"value"}

k := answer{}
// обратное действие: превращение json-строки в структуру
err = json.Unmarshal(bytes, &k) 
if err != nil { log.Fatal(err) }
```

Для потоковой работы с JSON есть `decoder` и `encoder`:

**Декодер `json.NewDecoder(io.Reader)`**:
Позволяет прочитать JSON из интерфейса `io.Reader` (из файла, например) в структуру с помощью метода `err := decoder.Decode(&structure)`. За раз читает из файла только одно json-значение (например, если в файле такой json: `[1,2,3]{"id":1}{"id":2}null"hello"`, то первый вызов метода прочитает `[1,2,3]`)

```go
// file - открытый json-файл

var user User
decoder := json.NewDecoder(file)

if err := decoder.Decode(&user); err != nil {
	log.Fatal("Error decoding JSON:", err)
}

fmt.Printf("Decoded user: %+v\n", user)
```

**Энкодер `json.NewEncoder(io.Writer)`**:
Позволяет преобразовать структуру в JSON и записать его в интерфейс `io.Writer` (в файл, например) с помощью метода `err := encoder.Encode(structure)`.

```go
// file - открытый файл для записи

user := User{ID: 1, Name: "John Doe", Email: "john@example.com"}
encoder := json.NewEncoder(file)

if err := encoder.Encode(user); err != nil {
	log.Fatal("Error encoding JSON:", err)
}
```

**Потоковое чтение списка json-объектов**:

```go
decoder := json.NewDecoder(file)

// читаем открывающую скобку массива '[' через Token()
if _, err := decoder.Token(); err != nil {
    log.Fatal(err)
}

// читаем элементы массива, пока не встретим ']'
for decoder.More() {
    var user User
    
    if err := decoder.Decode(&user); err != nil {
        log.Fatal(err)
    }
    
    fmt.Printf("Processed user: %+v\n", user)
}

// Читаем закрывающую скобку массива ']'
if _, err = decoder.Token(); err != nil {
    log.Fatal(err)
}
```

Для аналогичной работы с **YAML** в Go нужно поставить отдельный пакет `go-yaml`. Есть небольшое отличие в `yaml`-теках структур: можно указать `inline`, чтобы вложить одну структуру в другую:

```go
type HTTPConfig struct {
    Host string `yaml:"host"`
    Port int    `yaml:"port"`
}

type Config struct {
    HTTPConfig `yaml:",inline"`
    Debug      bool `yaml:"debug"`
}
```

>[!warning]
>1. Структуры с одинаковым полем, но с разным тегом, не равны между собой
>2. Тег никак не участвует в логике приложения, он нужен только для связи между форматами

## Переменные окружения

Еще одним способ хранения настроек для приложения являются переменные окружения. Они хранятся в формате `key=value`, где `value` является строкой. 
В Go можно получить значение переменной окружения через `os.Getenv(key)`, но если переменная не задана, то мы получим пустую строку. Более универсальный способ с проверкой существования - `val, ok := os.LookupEnv(key)`

Обычно переменные окружения для приложения хранят в текстовом файле `.env`. Чтобы загрузить их в Go и считывать как системные переменные окружения (т.к по умолчанию они ими не являются), есть библиотека `joho/godotenv`:

```go
import "github.com/joho/godotenv"

func main() {
    godotenv.Load() // Загружает переменные из .env в систему
    val, ok := os.LookupEnv("MY_VAR")
}
```

Если в окружении много переменных, то для их удобного получения есть библиотека `caarlos0/env`, она работает с тегами полей структуры:

```go
package main

import (
	"fmt"
	"log"
	
	"github.com/caarlos0/env"
    "github.com/joho/godotenv"
)

// определение структуры с тегами `env`
type Config struct {
    DBHost  string `env:"DB_HOST"`
    DBPort  int    `env:"DB_PORT"`
    // указываем значение по умолчанию
    APIKey  string `env:"API_KEY" envDefault:"12345678"` 
    Debug   bool   `env:"DEBUG"`
}

func main() {
    // Загружаем .env-файл (если он есть)
    if err := godotenv.Load(); err != nil {
        log.Println("No .env file found, relying on environment variables")
    }
    
    // Создаём экземпляр структуры
    cfg := Config{}
    
    // Парсим переменные окружения в структуру
    if err := env.Parse(&cfg); err != nil {
        log.Fatalf("Error parsing env: %v", err)
    }
    
    // Теперь переменные в структуре
    fmt.Printf("Config: %+v\n", cfg)
    fmt.Println("Database Host:", cfg.DBHost)
}
```

## Пайплайн сборки конфигурации для проектов

1. **Создаём единую структуру для хранения конфигурации**
2. **Прочитать файл конфига**
	Если указан файл конфига, то он считается первым источником истины. Данные оттуда стоит внести в созданную структуру.
3. **Применить env-переменные**
4. **Применить флаги**
	Всё-таки порой один-два флага при запуске программы мы используем. В таком случае надо не забыть о них.
5. **Использовать дефолтные значения**
	Для важных параметров, влияющих на работу системы, предусмотреть во всех случаях дефолтные значения.
6. **Провалидировать параметры конфигурации**
	Это важный шаг. Именно тут мы закладываем безопасность для наших параметров.
	Например:
	- проверка, что порт в допустимом диапазоне
	- таймауты положительные
	- URL и пути хотя бы не пустые, а лучше всего можно проверить, что они соответствуют определённому паттерну
	
	Приложение **должно** упасть как можно раньше, если с ним что-то не то. В данном случае мы должны упасть, если данные некорректны. Лучше завершиться с ошибкой, чем работать неправильно и подводить клиентов.
